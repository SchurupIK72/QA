# Шаблон баг-репорта

## Описание
Универсальный шаблон для фиксации багов на любом проекте.

## Обязательные поля (*)

### ID*
Уникальный идентификатор бага.
**Формат:** `BUG-[Section]-[№]`
**Формула Excel:** `="BUG-" & B43 & "-" & A43` (где A = №, B = Section)
**Примеры:** 
- `BUG-MainPage-1`
- `BUG-Career-2`
- `BUG-Footer-8`

### Section*
Раздел/модуль системы, где найден баг.
**Примеры:** Backend, Frontend, UI/UX, Database, API, Mobile, etc.

### Short prefix
Краткий префикс для группировки (опционально).

### Title*
Заголовок бага - должен отвечать на 3 вопроса: **Что? Где? Когда?**

**Правила:**
- Начинается с префиксов раздела и подраздела через точку
- Четко описывает проблему
- Содержит место возникновения

**Примеры:**
```
MainPage.Backend.Отсутствует обложка сайта (hero-секция) на Главной странице
Career.Frontend.Открывается страница 404 при нажатии на кнопку "Перейти к вакансиям"
Arena.Backend.При внесении изменений в БД данные не подтягиваются на клиент
```

### Steps (Шаги воспроизведения)*
Пошаговая инструкция для воспроизведения бага.

**Формат:**
```
1. Первый шаг (например: Открыть главную страницу сайта https://example.com)
2. Второй шаг (например: Пролистать до раздела "Карьера")
3. Третий шаг (например: Нажать на кнопку "Перейти к вакансиям")
4. Обратить внимание на [что конкретно проверяем]
```

**Правила:**
- Шаги должны быть максимально подробными
- Любой человек должен суметь воспроизвести баг по этим шагам
- Указывать конкретные значения, URL, параметры

### Actual Result (Фактический результат)*
Что произошло на самом деле после выполнения шагов.

**Примеры:**
- "Открывается страница 404 NOT FOUND"
- "Изображение не увеличивается при нажатии"
- "Кнопка не кликабельная"

### Expected Result (Ожидаемый результат)*
Что должно произойти согласно требованиям/ТЗ/логике.

**Примеры:**
- "Пользователь переходит в раздел 'Карьера'"
- "Изображение увеличивается при нажатии"
- "Открывается форма обратной связи"

### Environment (Окружение)*
Где был найден баг.

**Формат:**
```
Браузер/ОС и версия
или
Устройство, ОС, версия приложения
```

**Примеры:**
- `Google Chrome Version 141.0.7390.123 (Official Build) (64-bit)`
- `Windows 11, Chrome 141`
- `Android 15, Samsung A16`
- `iOS 17.2, iPhone 14 Pro`

### Build*
Номер билда, в котором обнаружен баг.

**Примеры:**
- `https://uvelka-petfood.tw1.ru/`
- `0.20.0 от 24.07.2025`
- `v1.3.45`

### Proofs (Пруфы)*
Ссылки на доказательства (скриншоты, видео, логи).

**Требования:**
- Скриншоты с выделенной проблемной областью
- Видеозапись воспроизведения бага
- Логи (при необходимости)
- Сетевые запросы (для API/Backend багов)

**Форматы файлов:** .png, .jpg, .mp4, .mov, .gif, .log

## Дополнительные поля

### Opening Date
Дата обнаружения бага.
**Формат:** `DD.MM.YYYY`

### Platform
Платформа: iOS, Android, Web, Windows, macOS, etc.

### Status
Текущий статус бага:
- `Open` - Новый баг, не взят в работу
- `In Progress` - В работе у разработчика
- `Reopen` - Переоткрыт (не исправлен или рецидив)
- `Done` - Исправлен и проверен
- `Fixed` - Исправлен, ожидает проверки
- `Archive` - Устарел/не актуален
- `No bug` - Отклонен (это не баг)

### Severity (Критичность)
- `Critical` - Блокирует систему/основной функционал
- `Major` - Серьезная проблема, есть обходной путь
- `Minor` - Незначительная проблема
- `Trivial` - Косметическая проблема

### Priority (Приоритет)
- `High` - Требует немедленного исправления
- `Medium` - Нужно исправить в ближайшее время
- `Low` - Можно исправить позже

### Developer (Исполнитель)
Имя разработчика, ответственного за исправление.

### Pre-condition (Предусловия)
Что нужно сделать/настроить перед воспроизведением бага.

**Примеры:**
- "Загрузить билд с русской локализацией"
- "Переключиться на английский язык"
- "Авторизоваться под логином test@mail.com"

### Additional info (Дополнительная информация)
Любая дополнительная информация:
- Комментарии от заказчика
- Связанные баги
- История исправлений
- Частота воспроизведения

### Bug frequency counter
Счетчик повторных обнаружений этого же бага.

## Контрольный чек-лист перед отправкой баг-репорта

- [ ] ID заполнен в правильном формате
- [ ] Заголовок отвечает на вопросы: Что? Где? Когда?
- [ ] Шаги воспроизведения подробные и понятные
- [ ] Фактический результат четко описан
- [ ] Ожидаемый результат соответствует ТЗ/требованиям
- [ ] Окружение указано полностью
- [ ] Прикреплены пруфы (скриншоты/видео)
- [ ] Выбран корректный статус
- [ ] Указана критичность и приоритет
- [ ] Баг можно воспроизвести по описанным шагам
